## F1. パレットの全地形種対応【ブロッカー】

### 問題

`flat_shading_spec.md` の実装メモの通り、現状 `palette.json` の `colors` は
grass / dirt / stone / cliff の**4種しか定義されていない**。
`stone_brick` / `bricks` / `cobblestone` などはファイル名ヒューリスティックで
この4種のいずれかにバケット分けされ、水・溶岩は暫定色がコードにハードコードされている。

これはつまり:

- **B案** は既存テクスチャの全バリエーションを表示する
- **A案** は4色に潰れた状態で表示される

という非対称な状態で比較されるということ。
A案が貧相に見えたとして、それが**ベタ色という方式の限界なのか、単に色を定義していないだけなのか**が区別できない。

`phase17-step1.md` の完了条件2「両案とも同一の面方向シェーディングを通っており、
**差分がアルベド供給元のみである**」が実質的に破られている。

### 対応

#### F1-1. 地形種の棚卸し

`assets/terrain/` 以下を走査し、検証シーンに出現しうる**全タイル種を列挙**する。
列挙結果を `docs/dev/terrain_tile_inventory.md` に一覧として書き出すこと。
（このリストは step2 以降でも使うため、成果物として残す）

#### F1-2. `palette.json` への base 色定義

列挙した全タイル種に対し、`colors` へ base 色を定義する。

- 水・溶岩も**必ず `palette.json` に定義**する。コードへのハードコードは全廃する
  （水・溶岩のシェーダー本体のフラット化は step2 のままでよいが、
  色定義の所在は今のうちに一本化しておく）
- 崖面が複数種ある場合は個別に定義する。`cliff` 1色に集約しない
- 色は暫定で構わないが、**互いに識別可能であること**を最優先に置く。
  A/B判断では「引いたカメラで隣接タイルが区別できるか」が争点になるため、
  ここで色が近すぎるとA案の評価が正しく行えない

`palette.json` を単一ソースとする原則は維持すること。

#### F1-3. ヒューリスティックの撤廃と明示マッピング

`flat_validation.gd` のファイル名ヒューリスティックによるバケット分けを削除し、
**タイルID → palette キーの明示的なマッピングテーブル**に置き換える。

**重要: 未定義タイルに遭遇した際、暗黙のフォールバックをしないこと。**

- 未定義タイルは **マゼンタ（`#FF00FF`）などの目立つエラー色**で描画する
- 同時に `push_warning()` でタイル名をログ出力する

今回の問題が判明しづらかった原因は、フォールバックが静かに成功していたことにある。
以後は画面上で即座に気付ける状態にする。

#### F1-4. 検証シーンへの未定義カウンタ表示

検証シーンのUIに `未定義タイル: N件` の表示を常設する。
`N > 0` の状態で A/B 判断を行ってはならない、という運用ルールを
`flat_shading_spec.md` にも明記すること。

## F2. 植生プロップのフラット化【ブロッカー】

### 問題

アルファカットアウトの草ビルボード（`grass_short` / `grass_tall`）が、
`flat_terrain_*.gdshader` が `ALPHA` を扱わないため変換対象から除外され、
元のPBRマテリアルのまま表示されている。

草プロップは地形の見た目への寄与が大きく、これが旧マテリアルのままでは
A/B どちらの評価も正確に行えない。

### さらに重要な点: 素直に実装すると破綻する

草プロップは 0° / 60° / 120° の3枚クロスクワッド構成である（`build_grass_props.py` 参照）。
ここに `face_factor()` を法線どおりに適用すると、**1つの草の房の中で3枚が別々の明るさになり、
房がバラけて見える。**

### 対応

#### F2-1. `flat_vegetation.gdshader` の新規作成

配置: `res://shaders/flat/flat_vegetation.gdshader`

```glsl
shader_type spatial;
render_mode unshaded, cull_disabled, depth_prepass_alpha;
#include "res://shaders/flat/flat_core.gdshaderinc"

uniform sampler2D albedo_tex : source_color, filter_nearest_mipmap;
uniform float alpha_threshold : hint_range(0.0, 1.0) = 0.5;
```

要点:

- **`cull_disabled` は必須。** クロスクワッドの裏面が消えるのを防ぐ
- **`face_factor()` を通さない。** 代わりに `face_top` を直接乗算し、
  **房を構成する3枚すべてを同じ明るさにする**。これが房を1つの塊として読ませる鍵
- `ALPHA = tex.a;` とし、`ALPHA_SCISSOR_THRESHOLD = alpha_threshold;` を設定。
  ピクセルアート由来のテクスチャなのでソフトアルファではなくシザーを使う
- 高さ方向カラーバンド（`apply_strata()`）は適用しない

#### F2-2. 株ごとの色ゆらぎ

地形タイルと同様の個体差を草にも出す。ただし**ピクセル単位でもクワッド単位でもなく、
株（インスタンス）単位**で揃える必要がある。

`vertex()` でインスタンス原点を取得し、`flat varying` で渡す:

```glsl
flat varying vec3 flat_instance_origin;
// vertex() 内:
flat_instance_origin = MODEL_MATRIX[3].xyz;
```

`fragment()` で `apply_variation(col, floor(flat_instance_origin.xz / tile_size))` を適用する。
これにより1株の3枚すべてが同一の色ゆらぎを受け、かつ隣の株とは色が違う状態になる。

#### F2-3. 変換対象への組み込み

`flat_validation.gd` の除外リストから草プロップを外し、
`flat_vegetation.gdshader` へ差し替える経路を追加する。

**A/B切替時の扱い**: 草プロップは A案 / B案 いずれでも `flat_vegetation.gdshader` を使う
（アルベドはテクスチャ由来のまま）。A/B の比較対象は地形タイルのみであり、
植生を切り替えると差分が増えて判断がぶれるため。この方針を実装コメントに明記すること。

## F3. `flat` 修飾子の付与

### 対応

`flat_core.gdshaderinc` の varying 宣言を以下に変更する。

```glsl
flat varying vec3 flat_world_normal;   // flat を付ける
     varying vec3 flat_world_pos;      // flat を付けない
```

### 理由

- **法線**: 面内で常に一定であるべき値。ボクセル地形は面が軸平行なので現状も破綻しないが、
  将来スムース法線を持つメッシュが混ざった際、面の内側にグラデーションが発生する。
  `flat` を付ければ原理的に防げる
- **座標**: 逆に補間が必須。タイル単位ハッシュと高さ方向カラーバンドが
  フラグメントのワールド座標に依存しているため、`flat` を付けると
  ポリゴン全体が単一のタイルIDに潰れて模様が消える。**絶対に付けないこと**

F2 で追加する `flat_instance_origin` も `flat` を付ける（株単位で一定であるべき値のため）。

## F4. `vertex()` セットアップのマクロ化

### 問題

`flat_shading_spec.md` の遵守事項4は「各シェーダーの `vertex()` に以下の2行を書くこと」
という規約になっている。この形の規約は、いずれ書き忘れが発生する。
しかも書き忘れた場合、シェーダーは**コンパイルが通ってしまい**、
varying が未初期化のまま描画されるため発見が遅れる。

### 対応

`flat_core.gdshaderinc` にマクロを定義する。

```glsl
#define FLAT_VERTEX_SETUP() \
    flat_world_normal = (MODEL_MATRIX * vec4(NORMAL, 0.0)).xyz; \
    flat_world_pos = (MODEL_MATRIX * vec4(VERTEX, 1.0)).xyz;
```

各シェーダーは以下のみを書く:

```glsl
void vertex() {
    FLAT_VERTEX_SETUP()
}
```

規約が「2行を正確に書く」から「マクロを1回呼ぶ」に変わり、
書き間違いが構文エラーとして表面化するようになる。

#### 注意

- Godot 4 のシェーダープリプロセッサは行継続（`\`）に対応しているが、
  問題が出る場合は1行の `#define` にまとめてよい
- varying を include 側で宣言しているため、`vertex()` を持たないシェーダーが
  この include を使うと未代入警告が出る可能性がある。出た場合は報告すること

### 併せて実施

`flat_shading_spec.md` の遵守事項4を、マクロ呼び出しの記述に書き換える。
