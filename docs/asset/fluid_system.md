# 流体システム設計ドキュメント（Fluid System Design）

位置づけ: 地形変化戦闘システムの拡張。マップ上の水・溶岩が地形の高低に応じて
流動し、盤面が時間とともに変化する仕組みを定義する。

ステータス: 設計確定（一部未確定事項あり。§9参照）

---

## 1. 設計方針（確定事項）

- **方式**: Minecraft式の距離減衰方式（高さ充填方式は採用しない）
- **減衰距離**: 水 = 6マス / 溶岩 = 5マス
- **侵食タイミング**: 敵味方問わず**1キャラの行動が終わるごと**に侵食tickを1回進める
- **侵食速度**: 1tickあたり 水 = 3マス / 溶岩 = 1マス
- **水源昇格**: あり（Minecraft準拠。流水が水平隣接する水源2つ以上で水源に昇格）
- **溶岩源昇格**: **なし**（溶岩は常に有限資源）

## 2. 用語定義

| 用語 | 意味 |
|---|---|
| 源（source） | 減衰の起点となる流体セル。水源 / 溶岩源 |
| 流れ（flowing） | 源から広がった流体セル |
| 強さ（strength） | そのセルから先へ広がれる残り距離 |
| 侵食tick | 流体の拡散・後退処理を1回進める単位。キャラ1人の行動終了ごとに1tick |
| フロンティア | 次のtickで新たに流体化する候補セルの集合 |

## 3. 拡散ルール

### 3.1 強さと減衰

- 源の強さ: 水 = 6、溶岩 = 5
- 流体セルは、水平隣接（4方向）の同じ高さのセルへ「自分の強さ − 1」の強さで広がる
- 強さ0のセルはそれ以上広がらない（存在はする）
- 結果として **水は源から最大6マス、溶岩は最大5マス** 届く

### 3.2 高低差の扱い

- 流体は**同じ高さ**の隣接セルへ広がる（等高拡散）
- 隣接セルが**低い**場合はそこへ落下し、**落下先で強さが最大値にリセット**される
  （水6 / 溶岩5。これにより落差さえあれば遠くまで流れる = Minecraft準拠）
- 隣接セルが**高い**場合は広がらない
- 流体はセル上面に乗る薄い層として扱い、**水位・深さの概念は持たない**
  （表示は既存のwater_plane / lava_planeと同じSURFACE_OFFSET方式）

### 3.3 水源昇格（無限水源）

- 流水セルが、**水平隣接する水源セルを2つ以上**持つとき、そのセルは水源に昇格する
- 昇格判定は侵食tick処理の中で行う
- 帰結（意図した戦術性）:
  - 海・湖などの大水域（全セル水源）に水路を接続すると、幅2以上の経路では
    水源化が連鎖し、減衰を超えて広がり続ける
  - 幅1の直線水路では昇格が起きず、理論どおり6マスで止まる
  - → 「大水域の近くでは地形変化スキルの影響が劇的になる」という
    **地形依存のスキル強度**が生まれる。マップの水域配置は戦略資源である

### 3.4 溶岩の非対称性

- 溶岩源昇格は**存在しない**。どれだけ溶岩源を並べても流れ溶岩は源にならない
- 溶岩は減衰5・侵食1マス/tickの遅い時限ハザードとして機能する
- （雰囲気付け）溶岩の拡散アニメーションは水より遅く・重く見せること

### 3.5 後退（源の喪失）

- 隆起スキルで源セルが埋まる等で源が失われた場合、その源に依存していた
  流れセルは消滅する
- 消滅も侵食tickで進める: 拡散と同レート（水3マス/tick、溶岩1マス/tick）で
  末端から後退させる（読みやすさの対称性のため）
- 実装Phase Aでは即時一括消滅で可（§8参照）。tick後退はPhase Bで対応

## 4. 侵食tickの処理順

1キャラの行動終了時に以下を1回実行する。

1. **整合性再計算**: 地形変化があった場合、全流体セルの強さをBFSで再計算し、
   到達不能になったセルを後退候補にする
2. **後退処理**: 後退候補を末端（強さの小さい順）から規定マス数だけ消滅させる
3. **拡散処理**: フロンティアから規定マス数だけ流体化する
   - 水: フロンティアをBFSで3層まで進める
   - 溶岩: 1層
4. **昇格判定**: 新規・既存の流水セルについて水源昇格を判定する
5. **相互作用判定**: 水と溶岩の接触を解決する（§5）
6. **ユニット影響の解決**（§9・未確定）

**決定性の保証**: 同一tick内で複数の拡散候補がある場合の処理順は
「源からの距離昇順 → グリッド座標順（y昇順→x昇順）」で固定する。
乱数を使わない。これによりプレビュー表示・AI評価・リプレイの結果が常に一致する。

## 5. 水×溶岩の相互作用（推奨案・要最終確認）

Minecraftの非対称ルールを継承する。

| 接触パターン | 結果 |
|---|---|
| 水が「流れ溶岩」セルに到達 | そのセルは**石（stone）地形に変化**し、流体は消滅 |
| 水が「溶岩源」セルに到達 | そのセルは**黒曜石相当の硬質地形に変化**（専用地形を作らない場合はstoneで可） |
| 溶岩が水セル（源・流れ問わず）に到達 | 溶岩側の進行セルが石化し、溶岩の進行が止まる |

- 石化したセルは通常の地面として歩行可能（「溶岩地帯に水路を引いて足場を作る」戦術の成立）
- 水系×土系など、属性ジョブ間のカウンター関係の基盤となる

## 6. UI・可読性要件（実装必須）

SRPGとして結果が予測できることを最優先とする。

- **スキルプレビュー**: 地形変化スキルのターゲティング時、その変化によって
  「最終的に流体が到達する全セル」を半透明表示する（tickを完全シミュレートした結果）
- **次tick予告**: 通常時、次の侵食tickで流体化するセル（フロンティア）を
  弱いハイライトで常時表示する（特に溶岩。1マス/tickなので予告が機能する）
- **ターン順UIとの連動**: 「あと何回行動が挟まると自セルが溶岩化するか」が
  ターン順リストから読み取れること（将来課題で可）

## 7. データ構造・アルゴリズム指針

```
MapCellData 追加フィールド:
  fluid_type: enum { NONE, WATER, LAVA }
  fluid_is_source: bool
  fluid_strength: int   # 水: 0..6 / 溶岩: 0..5

FluidSystem (新規クラス):
  recompute(map_data) -> 変更セルリスト      # BFS全再計算
  tick(map_data) -> FluidTickResult          # §4の1tick分
  preview_after_terrain_change(map_data, change) -> 到達セル集合  # UI用・非破壊
```

- 再計算は全水源・溶岩源をシードにしたマルチソースBFS。マップ20×20なら
  最悪でも数百セル走査で終わるため、性能上の考慮は不要
- `preview_after_terrain_change` は tick を内部で収束まで回した結果を返す
  （非破壊シミュレーション。MapDataのコピーに対して実行する）
- MapRenderer側: tickの結果セルにwater_plane/lava_planeを追加・削除する。
  拡散はBFS順に1セルあたり0.1〜0.2秒ずらして波及アニメーションさせる
