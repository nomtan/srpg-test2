# Godot SRPGプロトタイプ開発指示書

## 目的

Godotで、ボクセル風の立体マップ上で展開するSRPGのプロトタイプを作成したい。

目指す方向性は、Minecraftのような自由建築アクションではなく、
ブロック型ボクセルで構成されたジオラマ風マップを、SRPGの戦闘盤面として使うゲームである。

まずは以下を実現する。

- 立体的なボクセル風マップを表示する
- マップ上にSRPG用のグリッドを持つ
- ユニットを配置する
- カーソルでマスを選択する
- 移動可能範囲を表示する
- 選択したマスへユニットを移動させる
- 高低差を考慮した移動判定を行う

## 使用技術

- Godot 4.x
- GDScript
- 3Dプロジェクト
- まずはPC向け操作でよい
- 将来的にはiPhone / Android展開も視野に入れる


## ゲームの方向性

ジャンルはSRPG。

世界観はダークファンタジー寄り。
王国、帝国、エルフ領、精霊、瘴気、古代文明、神殿、砦、森、崖などが登場する戦術RPGを想定している。

ただし、今回のプロトタイプではストーリーやキャラクター演出は不要。
まずは戦闘マップとユニット移動の基礎を作る。

## 重要な設計方針

ボクセルマップをそのままMinecraftのように扱うのではなく、
見た目は3Dボクセル、ゲーム処理は2Dグリッド + 高さ情報として扱う。

つまり、内部的には以下のような構造にする。

```
{
  "x": 0,
  "z": 0,
  "height": 1,
  "terrain": "grass",
  "walkable": true,
  "move_cost": 1,
  "occupied_unit": null
}
```

- x, z はSRPG上のマス座標
- height はそのマスの高さ
- terrain は地形種別
- walkable は移動可能かどうか
- move_cost は移動コスト
- occupied_unit はそのマスにいるユニット

ボクセルのY座標を大量にSRPGロジックへ直接使うのではなく、
SRPG側では「そのx,zの最高到達面」をマスとして扱う。

## 最初に作るプロトタイプ範囲

最初のマップサイズは 8×8 でよい。

### マップ

以下のような簡易マップをコード上で生成する。

- 草地
- 土
- 石
- 高さ1〜3程度の段差
- 一部通行不可マス

例:
```
8×8の小さな立体マップ
一部に高さ2の高台
一部に高さ3の崖
一部に通行不可の岩
```

### ユニット

最初は味方ユニット1体だけでよい。

ユニット情報は以下を持つ。

```
class_name BattleUnit

var unit_name: String
var grid_x: int
var grid_z: int
var move_range: int
var jump_height: int
var team: String
```

初期値は以下でよい。

```
unit_name = "Vain"
grid_x = 1
grid_z = 1
move_range = 4
jump_height = 1
team = "player"
```

操作

- マウスまたはキーボードでカーソルを動かす
- カーソルが現在選択しているマスをハイライトする
- ユニットを選択できる
- ユニット選択時に移動可能範囲を表示する
- 移動可能なマスを選択するとユニットがそこへ移動する

操作は仮実装でよい。
完成UIは不要。

### 移動ルール

移動範囲計算では以下を考慮する。

- 上下左右の4方向移動
- 斜め移動は最初は不要
- move_cost を消費して移動する
- move_range を超えるマスには移動不可
- walkable=false のマスには移動不可
- occupied_unit があるマスには移動不可
- 高低差が jump_height を超える場合は移動不可

例:
```
現在の高さ: 1
隣の高さ: 2
jump_height: 1
=> 移動可能

現在の高さ: 1
隣の高さ: 3
jump_height: 1
=> 移動不可
```

###表示方針

#### マップ表示

最初は最適化よりも分かりやすさを優先してよい。

ただし、将来的にチャンクメッシュ化することを考慮し、
マップ生成ロジックと表示ロジックは分離する。

最初は以下のどちらかでよい。

- Cube Meshをマスごとに配置する
- または簡単なArrayMeshで表面だけ生成する

初期プロトタイプではCube配置でもよいが、コード上で後から最適化しやすい構成にする。

#### カメラ

SRPGらしい斜め見下ろしカメラにする。

- Orthographic Cameraを使用
- 斜め上から見下ろす
- マップ全体が見える位置に配置
- カメラ回転は最初は不要

#### ハイライト

以下の表示を用意する。

- カーソル中のマス
- 移動可能範囲
- 選択中ユニット

ハイライトは半透明のPlaneや簡易Meshでよい。

### シーン構成案

以下のような構成にする。

```
Main.tscn
├── VoxelMap
├── GridSystem
├── UnitManager
├── BattleCursor
├── Pathfinding
├── CameraController
└── UI
```

### スクリプト構成案

以下のファイルを作成する。

```
res://scripts/grid/GridCell.gd
res://scripts/grid/GridSystem.gd
res://scripts/map/VoxelMap.gd
res://scripts/unit/BattleUnit.gd
res://scripts/unit/UnitManager.gd
res://scripts/battle/BattleCursor.gd
res://scripts/battle/Pathfinding.gd
res://scripts/camera/CameraController.gd
res://scripts/Main.gd
```

### 各クラスの役割

#### GridCell.gd

1マス分のデータを表す。

持つ情報:

- x
- z
- height
- terrain
- walkable
- move_cost
- occupied_unit

#### GridSystem.gd

SRPG用グリッド全体を管理する。

責務:

- グリッド生成
- 指定座標のGridCell取得
- 座標が範囲内か判定
- ワールド座標とグリッド座標の変換
- マスの高さ取得
- マスの通行可否取得

#### VoxelMap.gd

見た目としてのボクセルマップを生成する。

責務:

- GridSystemの情報をもとに3Dブロックを配置
- terrainごとに見た目を変える
- 高さに応じてブロックを積む
- 将来的なメッシュ最適化に備えて、表示処理を分離する

#### BattleUnit.gd

SRPGユニットを表す。

責務:

- ユニット名
- 現在位置
- 移動力
- ジャンプ力
- チーム
- グリッド座標に合わせたワールド座標移動

#### UnitManager.gd

ユニット全体を管理する。

責務:

- ユニット生成
- ユニット選択
- 指定マスへの移動
- GridCellのoccupied_unit更新

#### BattleCursor.gd

SRPGカーソルを管理する。

責務:

- カーソル位置
- 入力処理
- 選択中マスのハイライト
- ユニット選択
- 移動先選択

#### Pathfinding.gd

移動可能範囲を計算する。

責務:

- BFSまたはDijkstraで移動範囲を計算
- move_costを考慮
- 高低差を考慮
- walkableを考慮
- occupied_unitを考慮

#### CameraController.gd

SRPG用のカメラを管理する。

責務:

- 斜め見下ろしカメラの配置
- 必要であればカメラ移動
- 最初は固定カメラでもよい

## 実装してほしい最初の成果物

以下が動く状態にしてください。

1. Godot 4の3Dプロジェクトで起動できる
2. 8×8の立体ボクセルマップが表示される
3. 斜め上から見下ろすSRPGカメラがある
4. マップ上にユニットが1体表示される
5. カーソルでマスを選択できる
6. ユニットを選択できる
7. ユニット選択時に移動可能範囲が表示される
8. 移動可能マスを選ぶとユニットが移動する
9. 高低差が大きすぎる場所には移動できない
10. 通行不可マスには移動できない
